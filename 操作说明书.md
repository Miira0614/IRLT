## 1. 系统能做什么

这个系统把 **LLM（大语言模型）** 当作"超级数据清洗工具"，自动帮你完成芯片需求的结构化处理。

### 核心功能一览

| 步骤 | 做什么 | 输入 | 输出 |
|------|--------|------|------|
| **第一步：智能分类** | AI 自动将每条需求分到 8 维 × 200+ 节点的分类目录中 | 原始需求 Excel | 带分类标签的需求 + 分类缓存 |
| **第二步：模板匹配** | 将需求文本与母版库中的标准模板匹配，提取变量参数 | 已分类的需求 | 带变量值的需求实例 |
| **第三步：拓扑补全** | AI 分析需求之间的 L1→L2→L3 追溯关系，补全缺失的层级 | 需求实例集合 | 完整追溯链 |
| **第四步：可视化导出** | 生成格式化的 Excel 预览报表 | 所有处理结果 | 格式化的 Excel 文件 |

### 你能用它做什么

- **场景1：新芯片需求整理** — 把一堆散乱的需求 Excel 扔进 `data/` 目录，跑一遍，得到结构化分类结果
- **场景2：需求追溯链建立** — 自动建立"客户需求→初始需求→系统需求"三层追溯关系
- **场景3：需求模板提炼** — 从具体芯片需求中提炼出可复用的参数化模板
- **场景4：需求质量检查** — 识别层级错标、分类不当、追溯断裂的需求

## 2. 环境准备（首次使用）

### 2.1 安装 Python

```bash
# 确认 Python 版本 >= 3.10
python --version
```

如果未安装，去 [python.org](https://www.python.org/downloads/) 下载 Python 3.10+。

### 2.2 安装依赖

打开终端（CMD 或 PowerShell），进入项目目录：

```bash
cd C:\Users\chenmo90239\Documents\PYTHON\New_IRLT_V3\New_IRLT
pip install -r requirements.txt
```

依赖清单（[requirements.txt](requirements.txt)）：

```
openai>=1.0.0       # LLM API 调用（支持 OpenAI 兼容接口）
openpyxl>=3.1.0     # Excel 文件读写
tqdm>=4.65.0        # 进度条显示
```

### 2.3 配置 LLM 连接

打开 `src/llm_client.py`，确认以下配置：

```python
# ==========================================
# 模型选择开关
# ==========================================
MODEL_CHOICE = os.getenv("MODEL_CHOICE", "local")  # 可选: "local", "siliconflow", "auto"
```

**三种模式说明：**

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| `"local"` | 使用公司内网 LLM 服务 | 公司内网环境（192.168.21.155:3000） |
| `"siliconflow"` | 使用硅基流动云 API | 外网环境，需要 API Key |
| `"auto"` | 优先本地，失败自动切硅基流动 | 网络不稳定时推荐 |

**如果使用 SiliconFlow 云 API，设置环境变量：**

```bash
# Windows PowerShell
$env:SILICONFLOW_API_KEY = "你的API密钥"
$env:MODEL_CHOICE = "siliconflow"

# 或者在 cmd 中
set SILICONFLOW_API_KEY=你的API密钥
set MODEL_CHOICE=siliconflow
```

### 2.4 验证环境

```bash
# 测试 API 连通性
python src/llm_client.py
```

看到 `[SUCCESS] API connectivity test passed!` 表示环境就绪。

### Step 0：规范配置检查 🆕（V4.1新增）

规范JSON是分类目录和初始模板的唯一数据源。处理需求前建议先检查配置一致性。

```bash
# 检查规范与当前配置是否一致
python main.py --check-spec

# 输出示例：
# [SPEC] 规范文件未变更 (上次同步: 2026-07-07T10:30:00)
# [SPEC] Categories: 7 L1 / 46 L2 / 156 L3
# [SPEC] Templates: 3211 total (existing: 3163, auto-gen: 48)
```

**如果提示规范文件已变更：**

```bash
# 方式1：启动时自动同步
python main.py --check-spec --auto-sync

# 方式2：独立同步工具（可预览差异后确认）
python script/sync_config.py --check      # 仅检查差异
python script/sync_config.py --sync       # 确认后同步
python script/sync_config.py --sync --force  # 跳过确认

# 方式3：仅同步 categories 或 templates
python script/sync_config.py --sync --categories-only
python script/sync_config.py --sync --templates-only
```

**规范书.doc 大版本更新时的完整流程：**

```bash
# 1. 从 Word 文档提取文本
python script/extract_doc_text.py "芯片需求结构化定义规范书V4.1.doc" -o spec_text.txt

# 2. 用 LLM 解析文本 → 更新规范JSON（或手动编辑）
#    （将 spec_text.txt 发给LLM，按规范JSON格式输出 → 替换原文件）

# 3. 从更新后的规范JSON同步配置
python script/sync_config.py --check
python script/sync_config.py --sync

# 4. 验证
python main.py --check-spec
```

### Step 1：准备数据文件

#### 情况A：你已经有了预处理好的 Excel（推荐）

直接把文件放到 `data/` 目录下：

```
data/
└── BMS_10 - CSE7580 - 完整需求追溯链.xlsx   ← 像这样
```

**Excel 文件命名规范：** `产品线_序号 - 芯片型号 - 文件类型.xlsx`
- 例如：`BMS_10 - CSE7580 - 完整需求追溯链.xlsx`
- 系统会从文件名自动解析"产品线"和"芯片型号"

**Excel 内容格式要求：**
- 第2个工作表（或唯一工作表）被使用
- C列（第3列）= 需求文本
- D列（第4列）= 需求序号/ID

#### 情况B：你只有原始需求包（需要先预处理）

原始需求包是指：客户需求、初始需求、系统需求分开放在不同 Excel 中。

1. 将原始文件放入 `raw_data/`：

```

├── CS32EX100 - 系统需求.xlsx
├── CPW3240 - 客户需求.xlsx
├── CPW3240 - 初始需求.xlsx
└── CPW3240 - 系统需求.xlsx
```

2. 运行预处理脚本：

```bash
python script/需求表格预处理.py
```

3. 预处理结果在 `raw_out/` 目录，文件名带有 `[聚合链矩阵]` 标记

4. 将 `raw_out/` 中的文件**复制到 `data/`** 目录

### Step 6：人工审核

系统采用 **Human-in-the-Loop** 设计，AI 处理完后需要人工复核：

1. **检查分类是否正确** — 打开预览 Excel，抽查分类结果
2. **检查层级判断** — L1/L2/L3 的层级标签是否合理
3. **检查追溯链** — 确认 L1→L2→L3 的父子关系正确
4. **检查变量提取** — 变量名称和数值是否正确

审核完成后，将已处理的文件移到 `data/已处理数据集/` 归档。

## 5. 各模块详细操作说明

### 5.1 需求表格预处理（`script/需求表格预处理.py`）

**什么时候用：** 当你拿到的原始数据是"客户需求.xlsx"+"初始需求.xlsx"+"系统需求.xlsx"分开的文件时。

**输入：** `raw_data/` 目录下的原始 Excel 文件

**输出：** `raw_out/` 目录下的 `[聚合链矩阵].xlsx` 文件

**操作步骤：**

```bash
# 1. 将原始文件按命名规范放入 raw_data/
#    命名规则：芯片型号 - 需求类型.xlsx
#    例如：
#      CS32EX100 - 初始需求.xlsx
#      CS32EX100 - 系统需求.xlsx

# 2. 运行预处理
python script/需求表格预处理.py

# 3. 将 raw_out/ 中的文件复制到 data/
cp raw_out/*[聚合链矩阵].xlsx data/
```

### 5.2 主程序（`main.py`）

**什么时候用：** 这是最常用的入口，99% 的情况下你只需要操作这个。

**命令行参数详解：**

```bash
python main.py [选项]

选项:
  --demo          演示模式：处理一条内置示例需求，验证系统是否正常
  --no-template   跳过模板匹配步骤，只执行分类（速度快但功能不完整）
  --data-dir DIR  指定输入目录（默认: data/）
```

**处理逻辑（自动执行）：**

```
对于 data/ 中的每个 .xlsx 文件：
  ├── 1. 读取 Excel，提取需求文本和ID
  ├── 2. 从文件名解析产品线和芯片信息
  ├── 3. 逐条调用 LLM 进行分类（三级目录匹配）
  ├── 4. 将分类结果与母版库进行模板匹配，提取变量
  ├── 5. 分析需求之间的追溯关系，补全缺失层级
  ├── 6. 生成格式化的 Excel 预览报表
  └── 7. 保存分类缓存和待定模板到 output/
```

### 5.3 Excel 修复工具（`fix_excel.py`）

**什么时候用：** 当 Excel 文件打不开、乱码、或 openpyxl 读取报错时。

```bash
python fix_excel.py  # 自动检测并修复
```

```json
    "details": { ... }
  }
}
```

**用途：** 如果重新运行同一个文件，系统会检查缓存，避免重复调用 LLM（省钱省时间）。

## 8. 常见问题排查

### Q1: 运行报错 `ModuleNotFoundError: No module named 'openai'`

```bash
pip install openai openpyxl tqdm
```

### Q2: 运行报错 `Failed to load category database`

检查 `data/config/categories.dbV4.1.json` 文件是否存在且格式正确。

```bash
# 验证 JSON 格式
python -c "import json; json.load(open('data/config/categories.dbV4.1.json', 'r', encoding='utf-8'))"
```

### Q3: LLM 调用失败 / 返回空结果

1. 检查网络连接
2. 检查 LLM 配置（`src/llm_client.py` 中的 `MODEL_CHOICE`）
3. 如果使用 SiliconFlow，检查 API Key 是否有效
4. 如果使用本地模型，确认 `http://192.168.21.155:3000/v1` 是否可访问
5. 尝试切换模式：`MODEL_CHOICE = "siliconflow"` → 或反过来

### Q4: Excel 读取不到数据

确认 Excel 格式：
- 需求文本在 **C 列**（第3列）或 **B 列**（第2列，在 `main.py` 中可调整）
- 需求 ID 在 **D 列**（第4列）或 **C 列**（第3列）
- 数据从 **第2行** 开始

如果列位置不对，修改 `main.py` 第36-37行：
```python
req_text_col = 2   # 改为你实际的需求文本列索引（从0开始）
req_id_col = 3     # 改为你实际的需求ID列索引（从0开始）
```

### Q5: 分类准确率低

1. 检查分类目录是否匹配你的需求类型（见 `data/config/categories.dbV4.1.json`）
2. 如果需求属于新的领域，可能需要扩展分类树
3. 查看 `Audit_Records/` 中的日志，找出分类失败的具体原因
4. 在 `src/categorization_module.py` 中调整置信度阈值（默认0.6）

### Q6: 处理速度太慢

1. 使用 `--no-template` 跳过模板匹配步骤
2. 切换更快的模型（如 `qwen3.6-flash`）
3. 减少一次处理的需求数量
4. 确认缓存目录 `output/classify/` 有效（已分类的需求不会重复调用 LLM）

### Q7: 上传 GitHub 前安全检查

**⚠️ 重要！** 代码中硬编码了 API Key，上传前必须处理：

```bash
# 1. 检查是否有敏感信息
grep -r "sk-" src/
grep -r "sk-" main.py

# 2. 将所有 API Key 替换为环境变量
#    在 src/llm_client.py 中，所有硬编码的 Key 应改为：
#    os.getenv("XXX_API_KEY", "")

# 3. 创建 .gitignore 文件（见附录）
```

### Q8: 🆕 规范JSON和categories.dbV4.1.json有什么区别？

**规范JSON** (`芯片需求结构化定义规范V4.0.json`) 是唯一数据源，包含：
- 完整的目录结构（维度 → 能力域 → 需求项）
- 每个需求项的 L1/L2/L3 完整需求文本
- 共156个需求项

**categories.dbV4.1.json** 是从规范JSON自动派生的**紧凑目录树**：
- 只有目录节点信息（id、name、level、parent_uid、children）
- 不包含L1/L2/L3的完整需求文本
- 发给LLM的只是这个紧凑树（~7KB），不是完整规范文本
- V4.1起由系统自动维护，无需手动编辑

### Q9: 🆕 规范书（.doc）更新了怎么办？

```bash
# 1. 提取文本
python script/extract_doc_text.py "芯片需求结构化定义规范书V4.1.doc" -o spec_text.txt

# 2. 将 spec_text.txt 发给 LLM，按规范JSON格式输出 → 更新 芯片需求结构化定义规范V4.0.json
#    （或手动在IDE中编辑 JSON 文件）

# 3. 同步配置
python script/sync_config.py --check
python script/sync_config.py --sync

# 4. 验证
python main.py --check-spec
```

### Q10: 🆕 会自动覆盖我手动修改的模板吗？

**不会。** 同步工具的策略是：
- **已有模板** → 完全不改动（保留人工维护的3163个模板）
- **新分类** → 自动从规范JSON的L1/L2/L3文本生成初始模板
- **孤立模板**（规范中已删除的分类）→ 标记但**不删除**
- **模板文本变更** → 不自动覆盖，需要人工确认

## 9. 附录

### A. 配置文件说明

#### 芯片需求结构化定义规范V4.0.json 🆕 — 规范定义（唯一数据源）

完整的芯片需求规范定义，包含所有需求项的目录信息和L1/L2/L3需求文本：

```json
[
  {
    "uid": "cat_de343t",
    "维度序号": "1",
    "维度": "核心功能",
    "能力域序号": "1.1",
    "能力域": "CPU",
    "需求项序号": "1.1.1",
    "需求项": "CPU规范定义",
    "原始需求": "面向工业/医疗/IoT终端产品...",
    "初始需求": "CPU内核基础框架维度...",
    "系统需求": "架构规格：内核版本、指令集..."
  }
]
```

**说明：** 共156个需求项，涵盖7大维度、46个能力域。categories.dbV4.1.json 从这里自动派生。

#### categories.dbV4.1.json — 分类目录数据库（V4.1起自动维护）

8大维度 × 46能力域 × 156三级分类节点的目录树。**V4.1起由 SpecDataProvider 自动从规范JSON派生，无需手动编辑。**

```json
{
  "categories": {
    "cat_xxx": {
      "uid": "cat_xxx",
      "id": "2.1",
      "name": "信号链-信号采集",
      "level": 2,
      "parent_uid": "cat_parent",
      "children": ["cat_child1", "cat_child2"],
      "description": "..."
    }
  }
}
```

**为什么发给LLM的只是这个紧凑树？**
- 分类时，LLM只需要知道目录的名称和层级关系，不需要看完整的L1/L2/L3需求文本
- 目录树只有 ~7KB / 209行，远小于完整规范JSON
- 模板匹配时才用到具体模板文本，且只发送当前分类的模板

#### Master_Requirement_Templates.json — 母版模板库

参数化的需求模板，包含变量占位符：

```json
{
  "tpl_001": {
    "template_id": "tpl_001",
    "level": "L3",
    "category_uid": "cat_xxx",
    "templates_text": "运行功耗<{{run_current_mA}}mA",
    "variables": [
      {"name": "run_current_mA", "type": "number", "unit": "mA"}
    ]
  }
}
```
